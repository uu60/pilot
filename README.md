# PILOT

**PILOT** stands for **Parallel In-path Lane-Oriented Triples**. It is a compact prototype derived from **Parsec** for evaluating a switch-assisted BMT delivery architecture for Boolean MPC.

PILOT is not intended to replace the full Parsec framework. Instead, it keeps the Parsec-style module organization and operator calling convention while removing features that are not needed for this experiment. The current focus is Boolean batch MPC, fixed-lane parallel execution, in-path BMT payload delivery, and a simplified secure database sort workflow.

## Background

In the original Parsec design, BMTs are treated as preprocessing material managed by the MPC runtime. PILOT explores a different architecture: BMTs are generated by a switch-like component on the communication path and appended to normal server-to-server messages.

The system uses four MPI ranks:

- `rank 0`: MPC server 0
- `rank 1`: MPC server 1
- `rank 2`: client and data holder
- `rank 3`: simulated switch

All server-to-server messages pass through the switch. During the Boolean AND `e/f` exchange, the switch forwards the original payload and appends a fixed-size BMT bundle for future AND operations. This makes BMT delivery passive and avoids adding extra BMT communication rounds.

## Architecture

PILOT uses fixed-lane parallelism. The maximum parallelism is configured at system initialization, and the number of BMT lanes is set to the same value. Each lane has one worker thread and one independent BMT stream.

Operators should be executed through the lane thread pool:

```cpp
auto future = LaneThreadPool::submit(laneId, [&] {
    BitwiseEqualOperator op(&xShares, &yShares, width,
                            SecureOperator::NO_CLIENT_COMPUTE);
    return op.execute()->_zis;
});
```

Within one lane, operators execute sequentially. Across lanes, operators execute in parallel. This avoids BMT ordering conflicts because each lane consumes only its own BMT stream.

## BMT Model

Only `BitwiseAndOperator` directly consumes BMTs. Higher-level operators such as Equal, Less, and Mutex are built from AND and do not manage BMTs themselves.

PILOT intentionally uses a simple width model:

- Supported bit width: `1..64`
- Each logical AND input consumes one 64-bit raw BMT
- If `width < 64`, only the low `width` bits are used
- The remaining bits are wasted by design

This avoids complex packed-BMT accounting and makes bundle usage deterministic.

## Build

PILOT uses CMake and MPI.

```bash
cd pilot
cmake -S . -B build
cmake --build build -j
```

## Run

PILOT requires four MPI processes because the switch is modeled as an additional rank.

```bash
mpirun -np 4 build/db/benchmark/pilot_db_sort \
  --rows=16 \
  --in_path_max_parallelism=4 \
  --in_path_bmt_bundle_size=64
```

For multi-column sort:

```bash
mpirun -np 4 build/db/benchmark/pilot_db_sort \
  --rows=16 \
  --multi_cols=1 \
  --in_path_max_parallelism=4 \
  --in_path_bmt_bundle_size=64
```

Important runtime options:

- `--in_path_max_parallelism=N`: number of fixed lanes and worker threads
- `--in_path_bmt_bundle_size=N`: fixed BMT bundle size per lane
- `--in_path_switch_rank=N`: switch rank, default is `3`

## Operator Usage

If the client holds plaintext inputs, use `Comm::CLIENT_RANK`:

```cpp
std::vector<int64_t> xs{1, 2, 3};
std::vector<int64_t> ys{1, 5, 3};
int width = 4;

BitwiseEqualOperator eq(&xs, &ys, width, Comm::CLIENT_RANK);
eq.execute()->reconstruct(Comm::CLIENT_RANK);

if (Comm::isClient()) {
    for (auto v : eq._results) {
        std::cout << v << " ";
    }
}
```

If the servers already hold secret shares, use `SecureOperator::NO_CLIENT_COMPUTE`:

```cpp
BitwiseAndOperator op(&xShares, &yShares, width,
                      SecureOperator::NO_CLIENT_COMPUTE);
auto zShares = op.execute()->_zis;
```

When using in-path BMTs, run operators that require AND inside a lane worker. The application layer only needs to manage lane assignment and batch size. BMT request, delivery, storage, and consumption are handled internally by the runtime, communication layer, switch simulator, and `BitwiseAndOperator`.

## Current Scope and Limitations

- Only Boolean batch MPC operators are supported.
- The switch is simulated as an MPI rank, not implemented as hardware.
- Parallelism is fixed at initialization time.
- Free-threaded operator execution is intentionally unsupported.
- Each lane is sequential; parallelism is across lanes.
- A single `BitwiseAndOperator` batch may not exceed the configured BMT bundle size.
- `BitwiseMutexOperator` consumes two AND streams, so its batch size should not exceed half of the BMT bundle size.
- Multi-column sort currently uses a packed comparison key and supports total key width up to 62 bits.

## Relationship to Parsec

PILOT inherits its coding style, module layout, and core MPC operator structure from Parsec. The main difference is that PILOT removes the general-purpose Parsec configuration space and specializes the runtime around the in-path BMT architecture. This makes it easier to reason about lane safety, BMT ordering, and switch-assisted delivery.
